# 会议室占用冲突排查与调度台

一个本地可运行的 Web 控制台应用，帮助行政人员高效管理会议室预约，自动检测时间冲突，支持改期、取消、房间锁定、黑名单管理等功能，并提供完整的操作日志和数据导出能力。

## 功能特性

### 核心功能
- **预约管理**：导入预约、查看预约列表、筛选条件（按房间、状态、组织者、日期）
- **冲突检测**：自动识别同一会议室的时间重叠，高亮显示冲突记录
- **冲突处理**：支持改期、取消预约记录，自动追踪冲突历史链
- **房间管理**：锁定/解锁会议室（仅管理员）
- **黑名单管理**：管理组织者黑名单，黑名单用户无法创建新预约
- **备注处理**：添加和编辑预约备注
- **历史日志**：记录所有操作历史（创建、更新、取消、改期、锁房、解锁、黑名单操作）
- **数据导出**：导出预约数据、房间数据、黑名单数据、操作日志和冲突历史
- **筛选持久化**：筛选条件自动保存，刷新或重启后自动恢复
- **冲突历史追踪**：取消和改期记录保留原始冲突链，可在历史列表、日志和导出中查看

## 使用说明

### 成功链路验证

#### 1. 冲突检测
- 打开控制台首页，系统已预置样例数据
- 会议室 A 有两条冲突预约（张三 09:00-10:00，李四 09:30-10:30）
- 冲突记录会以红色高亮显示，并标注冲突图标

#### 2. 改期操作
- 在控制台首页，找到冲突的预约记录
- 点击改期图标（铅笔图标），展开改期表单
- 输入新的时间（如 10:30-11:30），点击"确认改期"
- 改期成功后，预约状态变为"已改期"，冲突自动解除

#### 3. 取消操作
- 在控制台首页，找到待取消的预约记录
- 点击取消图标（垃圾桶图标），预约状态变为"已取消"
- 已取消的预约不再参与冲突检测

#### 4. 导出数据
- 导航到"数据导出"页面
- 点击各导出按钮，可分别导出预约数据、房间数据、黑名单数据和操作日志
- 导出格式为 CSV，可直接用 Excel 打开

#### 5. 房间锁定（管理员模式）
- 确保当前为管理员模式（左下角显示"管理员"）
- 导航到"房间管理"页面
- 点击房间卡片上的"锁定"按钮
- 房间状态变为"已锁定"，无法再创建新预约

#### 6. 黑名单管理
- 导航到"黑名单"页面
- 点击"添加黑名单"按钮，输入组织者姓名和原因
- 被加入黑名单的组织者无法创建新预约

## 核心问题修复：先改期再取消链路

### 问题描述
之前的版本在处理"先改期再取消"场景时存在缺陷：
1. 预约 A 和 B 在同一会议室时间重叠，产生冲突
2. 将预约 A 改期到无冲突的时间，冲突解除
3. 取消预约 A
4. **Bug**：取消记录中无法看出 A 曾经关联过冲突，也分不清 A 是先改期还是直接取消

### 问题根因
`cancelReservation` 函数只检查**当前**的冲突状态：
- 预约 A 改期后，`conflictId` 变为 `null`
- 取消时 `detectConflicts()` 找不到 A 在任何冲突组中
- 导致冲突历史记录丢失

### 解决方案
修改 `cancelReservation` 函数，优先检查历史冲突记录：
```typescript
// 检查当前冲突状态
const hasCurrentConflict = conflictGroup && conflictGroup.reservationIds.length > 1;
// 检查历史冲突记录
const hasHistoricalConflict = reservation.conflictHistory && reservation.conflictHistory.length > 0;
// 追溯历史冲突
const originalConflictId = reservation.originalConflictId || 
  (hasHistoricalConflict ? reservation.conflictHistory![0].conflictId : null);
```

### 三种取消场景的处理
1. **当前冲突取消**：预约仍在冲突组中，正常记录
2. **历史冲突追溯取消**：预约已脱离冲突，但通过 `conflictHistory` 追溯原冲突
3. **无冲突取消**：预约从未在冲突组中

### 数据结构增强
```typescript
interface ConflictHistoryEntry {
  conflictId: string;
  action: 'reschedule' | 'cancel';
  timestamp: string;
  operator: string;
  detail: string;  // 包含操作链信息
  relatedReservationIds: string[];
}
```

### 日志增强
- 改期操作：`[解除冲突: xxx]` 标记
- 取消操作：`[追溯历史冲突: xxx]` 或 `[关联当前冲突: xxx]` 标记
- 取消无冲突：`[无冲突记录]` 标记

### 导出数据增强
导出的 CSV 包含：
- **预约ID**：唯一标识
- **原冲突ID**：追溯原始冲突组
- **冲突历史链**：操作序列（如 `reschedule -> cancel`）
- **冲突历史详情**：完整操作记录

#### 1. 重叠预约互相覆盖（失败路径）
- 尝试将一条预约改期到与另一条预约重叠的时间
- 系统会提示"改期失败：新时间与其他预约冲突"
- 改期操作不会生效

#### 2. 普通用户无权锁房（失败路径）
- 切换为普通用户模式（左下角"切换为普通用户"）
- 导航到"房间管理"页面
- 尝试点击"锁定"按钮，系统提示"普通用户无权锁定房间"
- 锁定操作不会生效

#### 3. 改期后取消保留冲突链
- 对一条冲突预约进行改期操作
- 改期成功后再取消该预约
- 在操作日志中可查看完整的操作记录链
- 展开预约详情，可查看"冲突历史"标签页，显示改期或取消前的冲突组信息

#### 4. 筛选条件持久化
- 在控制台首页，设置筛选条件（例如：选择"会议室 A"，选择"待确认"状态）
- 刷新页面或按 F5 重新加载
- 观察筛选条件是否自动恢复为之前的设置
- 清除浏览器缓存后，筛选条件才会重置

#### 5. 跨会议室不误拦
- 创建两个不同会议室的预约，时间重叠
- 例如：会议室 A - 09:00-10:00，会议室 B - 09:30-10:30
- 这两个预约不会产生冲突，因为会议室不同
- 只有同一会议室的时间重叠才会被识别为冲突

#### 6. 冲突历史的导出与查看
- 导航到"数据导出"页面
- 点击"导出冲突历史"按钮
- 导出的 CSV 文件包含：预约ID、会议室、组织者、操作类型、冲突组ID、操作人、操作时间、关联预约IDs 等字段
- 预约数据导出中也包含"当前冲突ID"、"原冲突ID"和"冲突历史"字段

### 失败路径验证（详细步骤）

#### 1. 同一会议室改期冲突拦截
**目标**：验证改期到同一会议室的重叠时间会被拦截

**步骤**：
1. 定位到会议室 A 的两条冲突预约（张三 09:00-10:00，李四 09:30-10:30）
2. 点击李四的改期图标
3. 将时间改为 09:15-10:15（与张三的时间重叠）
4. 点击"确认改期"
5. **预期**：系统显示错误提示"改期失败：新时间与其他预约冲突"
6. **实际**：操作被拦截，改期不生效

**验证方式**：
- 查看浏览器控制台日志，应有相关错误输出
- 检查预约状态，李四仍然是"待确认"，时间未改变

#### 2. 管理员锁房权限验证
**目标**：验证只有管理员可以锁定房间

**成功路径**：
1. 确保当前为管理员模式
2. 导航到"房间管理"
3. 点击"会议室 A"的"锁定"按钮
4. **预期**：房间状态变为"已锁定"，按钮变为"解锁"
5. 尝试用该会议室创建新预约，系统提示"Room is locked"

**失败路径**：
1. 切换为普通用户模式
2. 导航到"房间管理"
3. 尝试点击"锁定"按钮
4. **预期**：`lockRoom` 函数返回 false，操作被拦截
5. 检查日志，应记录了权限不足的提示

#### 3. 冲突处理与历史链完整性
**目标**：验证冲突历史记录的完整性

**步骤**：
1. 定位冲突预约 A（张三 09:00-10:00）和 B（李四 09:30-10:30）
2. 点击预约 A 的改期图标
3. 将时间改为 10:30-11:30（无冲突）
4. **预期**：改期成功，冲突组重新计算
5. 展开预约 A 详情，查看"冲突历史"标签
6. **预期**：显示改期记录，包含冲突组ID、关联的预约ID
7. 导航到"操作日志"
8. **预期**：日志详情中包含"关联冲突: xxx"的标记
9. 取消预约 B
10. **预期**：取消成功，日志中记录原冲突组ID
11. 导出预约数据
12. **预期**：CSV 中包含"原冲突ID"和"冲突历史"字段

#### 4. 筛选条件持久化验证
**目标**：验证筛选条件在刷新后保留

**步骤**：
1. 在控制台首页设置筛选：会议室="会议室 A"，状态="待确认"
2. 打开浏览器开发者工具（F12）
3. 在 Console 中执行：`localStorage.getItem('room-scheduler-filters')`
4. **预期**：返回 JSON 格式的筛选条件，包含 roomId 和 status
5. 刷新页面（F5）
6. 检查筛选面板
7. **预期**：会议室和状态仍为之前的设置
8. 清除筛选（点击"清除筛选"按钮）
9. 刷新页面
10. **预期**：筛选条件为空（全部）

#### 5. 跨会议室时间重叠不产生冲突
**目标**：验证跨会议室的时间重叠不会被误判为冲突

**步骤**：
1. 创建会议室 A 的预约：10:00-11:00
2. 创建会议室 B 的预约：10:30-11:30
3. **预期**：两个预约都创建成功，无冲突提示
4. 检查控制台首页的"存在冲突"统计
5. **预期**：统计数字不变，未将跨会议室的重叠识别为冲突
6. 在预约列表中，这两个预约不应该以红色高亮显示

#### 6. 冲突历史的查询与追溯
**目标**：通过不同方式查询冲突历史

**通过预约详情查看**：
1. 找到有冲突历史的预约
2. 展开详情，查看"冲突历史"标签
3. **预期**：显示所有历史操作（改期/取消）、时间、操作人、冲突组ID

**通过操作日志查看**：
1. 导航到"操作日志"
2. 筛选"取消"或"改期"操作
3. **预期**：日志详情中显示"[关联冲突: xxx]"标记

**通过导出查看**：
1. 导航到"数据导出"
2. 导出"预约数据"，打开 CSV
3. **预期**：包含"原冲突ID"列，可追溯取消前的冲突组

**通过冲突历史导出查看**：
1. 导航到"数据导出"

3. **预期**：CSV 包含所有冲突历史的完整记录
